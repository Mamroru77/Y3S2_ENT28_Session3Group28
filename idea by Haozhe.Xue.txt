可以，而且从这门课的评分逻辑看，纯软件项目往往更稳。
Week 3–10 的主项目本来就允许你们做 mobile app、web platform、game、AI/Data product、consultancy + prototype，并且文件明确写了：你们不需要继续做 hardware/IoT。
更关键的是，老师评的是 process over technology：看你们有没有找真实用户、有没有根据证据迭代、文档写得专不专业、团队协作是否到位；甚至作业说明直接说，一个验证充分的 mobile app，可以比一个复杂但没测试好的 AI/硬件项目分更高。
所以如果你们不做硬件，我会说：完全没问题，甚至很可能是更聪明的选择。
前提是你们做的不是“空想软件”，而是一个真的能跑起来的 functional prototype；而且 slides 里还特别提醒了，不要用 Adalo、Softr 这种偷懒路线，必须是你们能解释清楚、能真实演示的 working prototype。
我建议你们按这条标准选题：
用户就在身边、第一轮访谈好找、核心流程 60 秒能讲清、8 周内能做出 end-to-end demo。 这也最符合 Week 3 brief、后续 Dev Log、Demo Day 和 Validation Report 的节奏。
我会优先给你们这 4 个纯软件方向：
1. 小组作业协作助手（最推荐）
给 XJTLU 学生用的 Web App，解决“分工混乱、截止时间忘记、组员沟通低效”的问题。
核心功能只做 4 个：创建项目、分配任务、deadline 看板、AI 帮你拆任务/生成 meeting summary。
这个题最好的一点是：用户特别好找，验证也特别容易。
2. 自习专注干预 App
不是做大而全的 productivity 平台，而是只解决一个问题：
“我一坐下就刷手机，怎么让我开始 25 分钟专注？”
核心流程可以是：选任务 → 开始番茄钟 → 中途干扰记录 → 结束反思。
这个题很适合做访谈和 before/after 测试。
3. 校园活动 / 社团匹配平台
帮学生快速找到“适合自己参加的活动、社团、比赛队友”。
可以做成 Web Platform，核心是：兴趣标签、活动推荐、组队发布、快速联系。
优点是场景真实，缺点是你们要把“和现有微信群/朋友圈相比有什么不同”讲清楚。
4. 学生二手教材 / 物品交换平台
聚焦一个小场景，比如“同校内教材和学习用品流转”。
核心流程：发布 → 搜索 → 联系 → 状态更新。
它很容易做成 working prototype，但你们要注意别做得太普通，最好加一个明确的小亮点，比如课程匹配、学院筛选、面交点推荐。
如果你问我哪个最适合冲分，我会排：
第一：小组作业协作助手
第二：自习专注干预 App
第三：校园活动/组队平台
原因不是它们“听起来更高级”，而是它们最符合这门课要的东西：
问题真实、用户近、测试快、迭代证据容易积累。 这对 Dev Log、Validation Report 和 Demo Day 都最友好。
你们现在如果决定走纯软件，我建议立刻定这 3 件事：
第一，定一个非常具体的用户痛点，不要写“帮助大学生提高效率”这种空话。
第二，只定 4 个 must-have features，因为 Week 3 brief 本来就要求你们这样定义范围。
第三，从这周开始就想好你们要找哪 5–10 个真实用户做验证，因为这门课后面分数很大一部分都靠证据链。
我建议你们现在直接把方向定成：
“面向 XJTLU 学生的小组作业协作助手 Web App”。
